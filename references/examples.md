# PRD E2E Test Agent Examples

Use these examples as the quality bar for generated test cases. Do not copy the domain blindly; adapt the actors, UI labels, data names, database fields, and external-system checks to the user's PRD and UI.

## Example 1: External API Asset Creation

Context:
- PRD says users can create an ability and select API as a service method.
- The UI has a workbench, an asset management ability page, a create ability drawer, a service-method step, a service-content step, and a finish step.
- Product database stores an external API group reference.
- The API asset source of truth is an external API hub.

### Executable Test Case

```markdown
### API-E2E-001 P0 Create ability with API service method

Purpose:
Validate that creating an ability through the UI with API selected creates a product ability record, creates one API hub group, and imports the OpenAPI interfaces completely.

Actor:
- Account: employee_a@example.test
- Role/permissions: Employee, owner of API Test App
- Tenant/domain: Tenant A

Preconditions:
- Product state: U-001 can open Workbench and has access to Asset Management > Ability.
- External system state: API hub test environment is reachable and has no group for this run ID.
- Required feature flags/config: API hub integration enabled in test environment.

Test Data:
- Generated data: ability name `E2E-AbilityAPI-202606231430-A7K2`
- Files: F-001 `openapi-ability-api-a.yaml`, expected endpoint count = 3
- Records to create: one ability via UI
- Data that must be provided by user: product URL, DB read access, API hub read token

UI Operation Path:
1. Open the product URL.
2. Log in as employee_a@example.test.
3. Open `Workbench`.
4. In the left navigation, open `Asset Management` > `Ability`.
5. Click `Create Ability`.
6. In the basic information step, fill ability name with `E2E-AbilityAPI-202606231430-A7K2`.
7. Select category, domain, source app `API Test App`, open scope, description, and tags according to the PRD field rules.
8. Click `Next`.
9. In the service-method step, select `API` and leave unrelated methods unselected unless the case says otherwise.
10. Click `Next`.
11. In the service-content step, upload `openapi-ability-api-a.yaml`.
12. Confirm the API card shows uploaded state and the interface preview shows 3 endpoints.
13. Confirm or fill API name, version, tags, and introduction.
14. Click `Next`.
15. In the documentation step, choose online Markdown and fill a short usage document.
16. Click `Finish Create`.
17. Confirm the finish page shows creation success.
18. Return to the ability list and confirm one new row exists for `E2E-AbilityAPI-202606231430-A7K2`.

Independent Verification:
1. Query cap-market ability table by ability name `E2E-AbilityAPI-202606231430-A7K2`.
2. Verify exactly one product ability record exists.
3. Verify `service_methods` contains `API`.
4. Verify `apihub_group_id` is not empty.
5. Query API hub group detail by `apihub_group_id`.
6. Verify the API hub group exists and is associated with this ability or its generated group name.
7. Query API hub interface list by `apihub_group_id`.
8. Verify method, path, and summary match the OpenAPI file.
9. Verify imported interface count is 3.
10. Query API hub groups by generated ability/group key and verify no duplicate group exists.

Expected Results:
- UI: creation succeeds and the ability appears in the ability list with API service method.
- Product database: one ability record exists, status is initial completed/unpublished state, and `apihub_group_id` is populated.
- External system: one API hub group exists and contains exactly 3 imported interfaces.
- Side effects: no duplicate product ability or API hub group is created.

Evidence To Capture:
- Screenshot of the service-content step showing uploaded OpenAPI and 3 endpoint previews.
- Screenshot of the ability list row after creation.
- Product ability ID and `apihub_group_id`.
- API hub group query response excerpt.
- API hub interface list response excerpt with count = 3.

Cleanup:
- If passed, delete/unpublish the ability according to environment cleanup rules.
- If failed or blocked, preserve product and API hub data for inspection.

Blocking Decision Rule:
Stop and ask the user if the UI cannot create any ability or if API hub synchronization fails for all API creation cases, because this blocks most integration E2E cases.
```

## Example 3: Bad vs Good Case Detail

Bad case detail, too vague:

```markdown
1. Create an API ability.
2. Check database.
3. Check API hub.
Expected: success.
```

Good case detail:

```markdown
1. Open Workbench > Asset Management > Ability.
2. Click Create Ability.
3. Fill the exact ability name and required fields.
4. Select API in the service-method step.
5. Upload the exact OpenAPI file and confirm the endpoint preview count.
6. Finish creation through the UI.
7. Confirm the list has exactly one new row for the generated ability name.
8. Query the product database by ability name and verify `apihub_group_id` is non-empty.
9. Query API hub by `apihub_group_id` and verify the group exists.
10. Query API hub interfaces and verify the count and paths match the OpenAPI source file.
Expected: UI, product DB, and API hub are mutually consistent.
```
