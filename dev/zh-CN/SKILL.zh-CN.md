# E2E Test Skill 中文版

> 说明：这是 `SKILL.md` 的中文阅读版，不作为 Codex/Agent 的正式 Skill 入口。正式入口仍以 `SKILL.md` 为准。

## 运行规则

默认只进行**用例设计**。创建一份专用 E2E 测试计划文件，然后停止，等待用户评审。只有在用户确认或修改计划，并且明确要求执行 E2E 后，才开始运行测试。

生成的计划、总结和评审请求要跟随用户语言输出。用户用中文时，正文用中文；用户用英文时，正文用英文。文件名、路径、用例 ID、代码 token 和精确 UI 文案保持原样。

使用三个用户决策节点：

1. **用例评审节点**：用户评审用例质量、PRD 覆盖完整性、高风险标记、假设和数据需求，用于决定是否执行 E2E。不要把它表述成最终上线验收，也不要暗示执行环境已经就绪。
2. **阻塞 BUG 节点**：执行中遇到阻塞当前用例、影响大量后续覆盖，或让必要验证不可靠的 BUG 时，停止让用户决策。
3. **结果与修复节点**：执行后展示结果、BUG、证据、清理状态和残余风险，让用户决定修复、延期、重跑或查看修复后结果。

在用例评审节点前，必须把计划顶部做成 **PM 审核包**。审核包必须包含简洁的决策摘要、校验脚本状态、必要时的二轮复核状态、覆盖缺口、高风险项、阻塞项，以及当前 1-3 个需要用户拍板的事项。详细用例是支撑材料，不要求用户第一次决策时逐条阅读。

对于 PRD 支撑的完整验收计划，或可执行用例超过 10 条的计划：

- 在请求评审前运行 `scripts/validate_plan.py`，先修复结构性失败。
- 在交给用户前，通过可用的 multi-agent 工具拉起独立 reviewer 子 agent。只给它 PM 审核包、决策摘要、覆盖审计、mock/数据计划、阻塞项和 3-5 个最高风险用例。
- reviewer 的必须修改项要先修复，阻塞结论要移入用户决策；对用户只摘要展示结论、必须修改项、用户决策、理由和主 agent 处理结果。

对于有 PRD 支撑的请求，默认追求 **完整 PRD 覆盖**，而不是代表性 happy path。PRD 中明确的必须项、验收标准、角色权限、业务规则、状态流转、异常/空状态、数据一致性规则和非功能/SLA 项，都必须映射为已覆盖、部分覆盖、阻塞、超出 E2E/专项或超出发布范围。如果用户只要求窄范围切片，要明确标注为聚焦切片，并说明未覆盖范围。

高风险项必须和覆盖状态分开标记。风险说明业务影响和执行优先级，不是覆盖状态。

保持 UI 操作和验证相互独立：

- **UI 操作路径**：像真实用户一样，通过可见页面、导航、表单、抽屉、上传、确认和状态变化操作。
- **独立验证**：完成 UI 操作后，通过数据库、API、日志、队列、文件、后台系统或外部系统验证持久化或集成结果。

不要用直接调用 API、写数据库、操作隐藏 DOM、派发合成 DOM 事件、写入 storage/session、调用组件内部方法，或使用自动化专用捷径来替代被测 UI 操作。每个被测动作都必须从真实用户可见入口开始；如果没有可见路径，就把用例标记为阻塞或失败。

这不是全自动测试工具。遇到依赖缺失时，先判断当前项目能否安全发现、创建或 mock。无法诚实解决时，用非技术人员能看懂的话说明缺口、影响和下一步决策。不能伪造证据、凭证、权限、环境或验证来源。

## 必要输入

必须具备：

- PRD 或等价产品需求。
- 可视化 UI 来源：截图、原型、HTML、运行中的产品，或能暴露真实操作路径的前端代码。
- 如果用户是非技术产品经理，先把目标转译成简短测试 brief：目标功能、PRD 来源、UI 来源、角色、环境、允许的测试数据操作，以及必须独立验证的内容。

如果需要执行真实测试，还需要提供或发现：

- 产品 URL 或本地运行方式。
- 测试账号、角色、租户、权限，或创建/mock 它们的权限。
- 用于独立验证的数据库、API、日志或外部系统访问能力。
- 创建、编辑、删除和清理测试数据的权限边界。

规划阶段，如果 PRD 提供了足够线索，可以把缺失的非敏感账号、角色、租户、权限、应用、文件和产品记录推导为 mock 数据。在环境能够创建或提供之前，这些推导数据只能视为执行需求。用户修改用例、角色、范围、文件或验证预期时，必须同步更新同一文件中的统一 mock/数据计划。

用户明确要求执行 E2E 后，先刷新 URL、账号矩阵、验证访问和清理权限。如果验证路径仍然空泛、过期、不可访问或不够独立，必须先更新计划，再开始执行。

## 工作流

1. **发现**
   - 阅读 PRD 和 UI 来源，识别参与者、对象、状态、规则、操作路径和集成点。
   - 在询问用户前搜索工作区中的依赖线索。完整发现顺序见 `references/workflow.md`。
   - 总结已发现内容、仍缺内容，以及哪些可以安全 mock 或创建。

2. **分析**
   - 提取业务能力、状态流转、权限、事实源系统和验收标准。
   - 对比 PRD 行为和 UI 能力。
   - 建立 PRD 覆盖审计；当矛盾、不可达状态、预期结果缺失或数据归属不清会影响套件时，停止澄清。

3. **设计**
   - 按业务能力或集成边界组织用例。
   - 每个可执行用例都包含参与者、前置条件、数据准备、精确 UI 路径、独立验证、预期结果、证据、清理方式和阻塞规则。
   - 按用户语言使用对应模板，把套件写入一份专用 Markdown 文件。
   - 把 PM 审核包放在详细覆盖和用例之前。
   - 要求用户评审或修改文件，然后再执行。

4. **用户批准后才执行**
   - 刷新执行就绪状态和独立验证来源。
   - 严格按照已批准的 UI 路径执行。
   - API/数据库只能用于已声明的数据准备，或 UI 操作后的独立验证。
   - 采集证据，包括为已通过的 UI 用例主动保存 UI 截图；把结果追加到同一份计划文件，并应用 `references/execution-guardrails.md`。
   - 执行结束后，如已有证据文件，运行 `scripts/validate_evidence.py "{plan_path}"`；缺少必要 UI 证据或独立验证证据的通过用例要降级为无法确认。

5. **总结**
   - 在同一文件中更新通过、失败、阻塞、跳过数量，证据引用，缺陷，修复情况，延期 BUG，清理状态，残余风险和仍需用户决策的事项。

## 参考文件路由

- 在生成测试计划、测试用例、评审请求、依赖请求、阻塞 BUG 问题或运行总结前，按用户语言读取对应模板：英文输出用 `references/templates.md`，中文输出用 `dev/zh-CN/references/templates.zh-CN.md`。
- 在生成或评审任何 PRD 支撑的完整验收计划时，或者用户询问覆盖是否完整时，按用户语言读取对应覆盖文件：英文输出用 `references/coverage.md`，中文输出用 `dev/zh-CN/references/coverage.zh-CN.md`。
- 当需要更详细的依赖发现、测试数据规划、评审门禁、执行就绪检查或结果分类时，按用户语言读取对应工作流文件：英文输出用 `references/workflow.md`，中文输出用 `dev/zh-CN/references/workflow.zh-CN.md`。
- 当请求涉及外部系统验证、多角色行为，或生成的用例可能过于模糊时，读取 `references/examples.md`。
- 在执行已批准用例，或判断遇到 BUG 后是否继续前，读取 `references/execution-guardrails.md`。

## 文件布局与脚本

- 如果用户没有指定路径，默认把计划写入 `e2e-test-plans/`。
- 测试计划文件名使用 `e2e-test-plan-{feature}-{run_id}.md`。
- 证据放到 `e2e-test-evidence/{run_id}/`。
- 需要时运行 `python3 scripts/create_run_id.py "{feature}"` 生成 run ID。
- 需要时运行 `python3 scripts/scaffold_plan.py "{feature}" --run-id "{run_id}"` 创建计划和证据目录。
- 在请求评审或执行前运行 `python3 scripts/validate_plan.py "{plan_path}"`；如果缺少参与者、UI 路径、独立验证、证据、清理、PRD 覆盖审计、决策摘要、自动门禁或二轮复核，先修正后再继续。
- 执行结束后运行 `python3 scripts/validate_evidence.py "{plan_path}"`，确认已通过的 UI 用例在声明的证据目录下同时具备截图证据和独立验证证据。

## 输出模式

选择与用户请求最匹配、范围最窄的模式：

- **仅用例设计**：创建专用测试计划文件，然后停止等待评审。
- **执行计划**：创建可运行用例、数据需求、凭证清单和风险门禁。
- **运行执行**：执行已批准用例，收集证据，遇到阻塞 BUG 时停止，并追加结果。
- **运行后报告**：总结已有运行日志，优先更新同一个文件。

默认使用 Markdown。只有在对机器执行有帮助时，才使用 JSON 或 YAML。
