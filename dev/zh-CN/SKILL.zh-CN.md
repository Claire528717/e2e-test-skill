# E2E Test Skill 中文版

> 说明：这是 `SKILL.md` 的中文阅读版，不作为 Codex/Agent 的正式 Skill 入口。正式入口仍以 `SKILL.md` 为准。

## 默认行为

默认只进行**用例设计**。创建一个专门的 E2E 测试计划文件，然后停止，等待用户评审。

只有在用户确认或修改测试计划，并且进一步明确下达执行指令后，才开始执行测试。

使用三个用户决策节点：

1. **用例评审节点**：用户评审测试用例质量、PRD 覆盖完整性、高风险标记、假设和数据需求，用于决定是否执行 E2E 测试。不要把这个节点表述成最终上线验收，也不要暗示执行环境已经就绪。
2. **阻塞 BUG 节点**：执行过程中，如果阻塞 BUG 阻止当前用例、影响大量后续覆盖，或让必要的独立验证不可靠，就停止并让用户决定是否先修复再继续。
3. **结果与修复节点**：执行结束后，向用户展示结果、BUG、证据、清理状态和残余风险，由用户决定是否修复全部 BUG、延期部分 BUG、重跑失败/阻塞用例，并查看最终修复结果。

对于有 PRD 支撑的请求，默认追求 **PRD 覆盖完整性**，而不是只做代表性 happy path。先从 PRD 中抽取明确必须项、验收标准、角色权限、业务规则、状态流转、异常/空状态、数据一致性规则和非功能/SLA 项，形成覆盖审计。每个 PRD 项都必须映射到以下之一：

- 已被可执行 E2E 用例覆盖。
- 部分覆盖，并说明缺失分支。
- 因缺少决策、账号、环境、系统或验证来源而阻塞。
- 超出 E2E 范围，需要性能、安全、压测等专项测试。

如果用户只要求窄范围切片，要明确标注这是聚焦切片，并简要说明未覆盖的 PRD 范围。不要把切片计划包装成完整发布验收计划。

高风险 PRD 项必须和覆盖状态分开标记。高风险项仍然可以是已覆盖、部分覆盖、阻塞或专项；风险级别说明业务影响和执行优先级，不等同于覆盖状态。

这不是 100% 自动化测试工具。遇到依赖缺失时，先判断能否从当前项目中合理发现、创建或 mock；如果无法诚实解决，要用非技术人员能看懂的话说明缺什么、影响什么、下一步需要用户决定什么。不能绕过 UI、伪造证据，或假装缺失的环境、账号、权限、验证来源已经存在。

如果用户请求中没有包含账号、角色、租户或权限信息，可以根据 PRD 推导一份 mock 测试主体矩阵用于规划。本地测试执行阶段，默认这些主体大概率需要 mock 或创建，除非用户已经提供真实测试主体。必须在计划中记录每个 mock 主体、角色、租户、权限、数据对象、来源、范围和清理规则。执行前必须把这些 mock 主体视为数据需求；除非当前环境能够创建或提供这些主体，否则不能假装它们已经存在。

只有当当前环境具备所需 URL、账号或可创建的测试主体、数据权限和验证访问能力时，才执行真实测试。

用户明确要求执行 E2E 后，先根据此时提供的 URL、数据库/API/日志/外部系统访问、账号矩阵和清理权限，刷新执行就绪状态和独立验证计划。如果某个验证路径仍然空泛、过期、不可访问或不够独立，必须先更新测试计划，再开始执行。

## 核心规则

保持 UI 操作和验证相互独立：

- **UI 操作路径**：像真实用户一样，通过可见页面、导航、表单、抽屉、上传、确认和状态变化进行操作。
- **独立验证**：完成 UI 操作后，通过数据库、API、日志、队列、文件、后台系统或外部系统验证持久化或集成结果。

不要用直接调用 API、写数据库、操作隐藏 DOM、派发合成 DOM 事件、写入 storage/session、调用组件内部方法，或使用自动化专用捷径的方式替代被测 UI 操作。每个被测动作都必须从真实用户也能使用的可见入口开始；如果没有可见路径，就把用例标记为阻塞或失败。

## 必要输入

必须具备：

- PRD 或等价的产品需求文档。
- 可视化 UI 来源：截图、原型、HTML、运行中的产品，或能暴露真实操作路径的前端代码。
- 如果用户是非技术产品经理，先把用户目标转译成一份简短测试 brief：目标功能、PRD 来源、UI 来源、角色、环境、允许的测试数据操作，以及必须独立验证的内容。
- 对依赖缺口、mock 假设、阻塞用例和需要用户决策的事项，使用非技术人员能看懂的话说明。

如果需要执行真实测试，还需要提供或发现：

- 产品 URL 或本地运行方式。
- 测试账号、角色、租户、权限，或创建它们的权限。
- 用于独立验证的数据库、API、日志或外部系统访问能力。
- 创建、编辑、删除和清理测试数据的权限边界。

在规划阶段，如果 PRD 提供了足够线索，可以推导缺失的账号、角色、租户、权限、应用和测试文件作为 mock 数据。只询问那些会阻塞规划或真实执行的缺失项。本地执行时，优先使用隔离的 mock 数据，并清楚记录 mock 范围，而不是要求非技术用户手工提供用户、角色、租户和权限记录。

优先让一次测试运行聚焦在一个功能切片或一个集成边界上。

## 工作流

1. **发现**
   - 阅读 PRD 和 UI 来源，识别参与者、对象、状态、规则、操作路径和集成点。
   - 从 must/shall/checklist、验收标准、角色权限矩阵、业务规则、状态机、异常/空状态、数据字段、集成契约和非功能需求中建立 PRD 覆盖清单。
   - 在询问用户前，先搜索工作区中的依赖线索：文档、路由、API client、`.env*`、mock、fixture、测试、schema/migration、OpenAPI/Swagger、SDK 和本地运行脚本。
   - 总结已经发现的内容和仍然缺失的内容。
   - 在询问用户前，判断每个缺失依赖能否通过项目文档、本地配置、安全 mock 数据或生成测试文件合理解决。
   - 如果用户请求执行测试，在承诺运行前先确认当前环境是否真的能进行 UI 交互和独立验证。

2. **分析**
   - 提取业务能力、状态流转、权限、事实源系统和验收标准。
   - 对比 PRD 行为和 UI 能力。
   - 将每个覆盖清单项标记为已覆盖、部分覆盖、阻塞或超出 E2E/专项范围。还没有映射到用例的 PRD 项要视为缺口，而不是默认已覆盖。
   - 当存在矛盾、不可达状态、预期结果缺失或数据归属不清，并且会影响测试套件时，停止并向用户提问。

3. **设计**
   - 按业务能力或集成边界组织用例。
   - 对完整发布或交付验收计划，覆盖所有 PRD P0/P1 和明确验收项。优先做高价值 P0/P1 深度覆盖，而不是低价值穷举，但不能遗漏 PRD 要求而不记录原因。
   - 对每个未完全覆盖的 PRD 要求，要么补用例，要么用具体原因标记为阻塞/范围外，并说明需要用户做什么决定。
   - 每个可执行用例都要包含参与者、前置条件、数据准备、精确 UI 路径、独立验证、预期结果、证据、清理方式和阻塞规则。
   - 使用 `references/templates.md` 将测试套件写入一个专门的 Markdown 文件。
   - 当用例可能变得空泛，或涉及外部系统验证时，使用 `references/examples.md` 作为质量标准。
   - 在请求评审前包含 PRD 覆盖审计和需求到用例矩阵，让用户不用逐条读用例也能看出这是完整覆盖还是刻意切片。
   - 执行前必须包含推导或已提供的测试主体矩阵，以及数据/mock 范围记录。
   - 用单独的风险字段或风险列标记高风险项，不要把风险混进覆盖状态。
   - 如果用户修改用例、角色、范围、文件或验证预期，必须同步更新同一文件中的 mock/数据计划和测试主体矩阵，再重新请求评审。
   - 要求用户评审或修改文件，然后再执行。

4. **用户批准后才执行**
   - 首条用例执行前，重新检查并更新独立验证来源。
   - 严格按照已批准的 UI 路径执行。
   - API/数据库只能用于已声明的数据准备，或在 UI 操作完成后的独立验证。
   - 在关键步骤后采集证据。
   - 除非用户要求单独报告，否则将结果追加到同一个测试计划文件中。
   - 应用 `references/execution-guardrails.md` 中的停止规则。

5. **总结**
   - 在同一个文件中更新通过、失败、阻塞、跳过数量，证据引用，缺陷，修复情况，延期 BUG，清理状态，残余风险和仍需用户决策的事项。

## 参考文件路由

- 在生成测试计划、测试用例、评审请求、依赖请求、阻塞 BUG 问题或运行总结前，读取 `references/templates.md`。
- 在生成或评审任何 PRD 支撑的完整验收计划时，或者用户询问覆盖是否完整时，读取 `references/coverage.md`；中文版参考为 `dev/zh-CN/references/coverage.zh-CN.md`。
- 当请求涉及外部系统验证、多角色行为，或生成的用例可能过于模糊时，读取 `references/examples.md`。
- 当任务横跨设计和执行，或需要更详细的依赖发现、测试数据规划、评审门禁、结果分类时，读取 `references/workflow.md`。
- 在执行已批准用例，或判断遇到 BUG 后是否继续前，读取 `references/execution-guardrails.md`。

## 默认文件布局

- 如果用户没有指定路径，默认把测试套件写入 `e2e-test-plans/`。
- 测试计划文件名使用 `e2e-test-plan-{feature}-{run_id}.md`。
- 将截图和其他证据放到 `e2e-test-evidence/{run_id}/` 下。
- 在整个运行过程中保持测试计划文件和证据路径一致。

## UI 保真度规则

- 每个被测用户动作都必须从真实用户也能看到和使用的入口开始，例如按钮、链接、菜单、表单字段、标签页、抽屉、弹窗、拖拽柄、开关、分页或确认控件。
- 自动化工具可以驱动浏览器，但不能通过真实用户无法访问的路径改变产品状态。
- 低层框架辅助方法只能用于实现已经由可见 UI 触发的浏览器能力；证据必须显示交互前的可见入口和交互后的可见结果。
- 如果产品没有提供可见路径，就将用例标记为阻塞或失败，不要通过隐藏 DOM、内部 selector、组件内部方法、localStorage/sessionStorage、网络调用或数据库写入来绕过。

## 脚本

当需要稳定的文件布局或格式校验时，使用内置脚本：

- 运行 `python3 scripts/create_run_id.py "{feature}"` 生成标准 run ID。
- 运行 `python3 scripts/scaffold_plan.py "{feature}" --run-id "{run_id}"` 创建默认测试计划文件和证据目录。
- 在请求用户评审或执行前，运行 `python3 scripts/validate_plan.py "{plan_path}"`；如果缺少 actor、UI path、独立验证、证据、清理信息或 PRD 覆盖审计，先修正后再继续。

## 输出模式

选择与用户请求最匹配、范围最窄的模式：

- **仅用例设计**：创建专门的测试计划文件，然后停止，等待评审。
- **执行计划**：创建可运行用例、数据需求、凭证清单和风险门禁。
- **运行执行**：执行已批准用例，收集证据，遇到阻塞 BUG 时停止，并追加结果。
- **运行后报告**：总结已有运行日志，优先更新同一个文件。

默认使用 Markdown。只有在对机器执行有帮助时，才使用 JSON 或 YAML。
