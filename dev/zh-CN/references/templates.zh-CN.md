# E2E Test Skill 模板

## 专用测试文件

整条测试生命周期只使用一份专用 Markdown 文件。推荐文件名格式如下：

`e2e-test-plan-{feature}-{run_id}.md`

如果用户没有指定路径，默认放在 `e2e-test-plans/` 下，证据放在 `e2e-test-evidence/{run_id}/` 下。

在执行前，这个文件是 E2E 测试计划。只有在用户明确授权执行并且运行开始后，才把执行结果、BUG 和清理状态追加到同一份文件里。

## PRD 覆盖审计

每个 PRD 支撑的完整验收计划都使用本节。聚焦切片计划也要包含简版，说明哪些范围是刻意不覆盖的。

```markdown
## PRD 覆盖审计

覆盖定位：
- 计划类型：完整发布验收 / 聚焦切片 / 仅原型 / 正式环境 E2E
- 覆盖声明：完整 PRD 覆盖 / 仅 P0-P1 / 仅选定流程
- 本 E2E 计划不覆盖：
- 需要专项测试：
- 受产品决策或环境阻塞：

覆盖汇总：
| 状态 | 数量 | 含义 |
|---|---:|---|
| 已覆盖 | 0 | 有可执行 UI 路径和独立验证 |
| 部分覆盖 | 0 | 缺少分支、角色、状态或验证来源 |
| 阻塞 | 0 | 依赖解决前不能诚实规划或执行 |
| 超出 E2E / 专项 | 0 | 需要性能、安全、压测、可访问性或其他专项 |
| 超出发布范围 | 0 | PRD 明确排除或延期 |

需求覆盖矩阵：
| PRD 项 | 验收点 | 风险 | 覆盖状态 | 用例 ID | 缺口 / 决策 |
|---|---|---|---|---|---|
| PRD-001 | {需求或 checklist 项} | 高 / 中 / 低 | 已覆盖 / 部分覆盖 / 阻塞 / 超出 E2E / 超出发布范围 | {CASE-ID} | {缺少的分支、依赖、专项测试或无} |

覆盖门禁说明：
- 每个已覆盖行都必须引用已存在的可执行用例 ID。
- 每行只能使用一种覆盖状态；高风险和阻塞说明放在风险或缺口/决策字段里，不放进状态字段。
- 每个高风险的部分覆盖行都必须说明缺失分支或验证来源，并补用例或说明执行时需要的决策。
- 每个阻塞行都必须说明缺少的决策、账号、环境、系统或权限。
- 不要在明确 P0/P1 PRD 项未映射时请求评审，除非它被标记为阻塞或超出 E2E 且原因明确。
```
