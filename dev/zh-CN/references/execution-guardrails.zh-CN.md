# 执行护栏

在执行已批准的 E2E 用例，或者决定在遇到 BUG 后是否继续之前，先读这份参考。

## 捷径规则

- 保持 UI 保真：每个被测用户动作都必须从真实用户也会使用的可见入口开始，例如按钮、链接、菜单项、表单字段、标签页、抽屉、对话框、上传/拖放区、拖拽把手、复选框、单选框、开关、分页控件或确认控件。
- 自动化 API 可以驱动浏览器，但不能通过真实用户无法走到的路径改变产品状态。不要修改 DOM 状态、调用组件方法、派发合成产品事件、直接改本地存储/session，或用后端 API 替代正在测试的 UI 动作。
- 只有当低层框架辅助方法是在完成可见 UI 动作之后，用来实现用户触发的浏览器能力时，才允许使用。证据必须展示交互前的可见入口和交互后的可见结果。
- 如果产品没有提供所需动作的可见路径，就把用例标记为阻塞或失败。不要通过隐藏 DOM、内部选择器、组件内部方法、localStorage/sessionStorage、网络调用或数据库写入绕过去。
- 不要为了创建被测对象而直接调用后端 API；如果用例测试的是通过 UI 创建对象，就必须通过 UI 创建。
- 不要用数据库写入来强行推进 UI 本应完成的状态流转。
- 不要只凭本地产品记录推断外部集成成功；要去查询外部系统。
- 不要只凭外部系统记录推断产品数据一致；也要查询产品数据库。
- 对于涉及持久化或集成的用例，不能只凭成功 toast 就宣称通过。
- 不要用一个高权限账号替代角色切换测试。

## 生产安全

- 默认使用本地、测试或预发环境。
- 如果目标 URL 或应用上下文看起来像生产环境，而用户没有明确批准写入操作，就停止并询问。
- 在生产环境中，除非用户明确授权了精确的写入范围，否则只做只读验证。
- 不要在未确认环境和影响的情况下，创建、更新、删除或清理共享生产数据。
- 不要把真实客户、财务、凭证或隐私数据当作 mock 数据。

## 证据规则

在每个主要步骤之后记录证据：

- 当前页面、抽屉、弹窗、标签页或可见状态。
- 已提交的值。
- 有条件时截屏。
- 请求、关联、产品记录、数据库和外部系统 ID。
- 相关数据库/API/日志响应摘录。
- 清理状态，或者说明为什么暂缓清理。

如果某个用例缺少关键证据，就把它标记为无法确认，而不是通过。

证据必须包含足够的可见 UI 状态，证明确实使用了面向用户的路径：交互前的可见入口、必要时交互时的可见提交值或选择状态，以及交互后的可见结果。只靠隐藏 DOM 状态、内部选择器、自动化 API、网络调用，或单独的数据库/API 验证，不能证明 UI 操作路径成立。

## BUG 处理

默认情况下，记录 BUG，但不要自行修复。

如果 BUG 出现以下情况，就停止并询问用户，不能绕过或继续：

- 阻塞当前测试步骤。
- 阻塞当前用例。
- 会让后续大量用例变得不可靠。
- 影响必要的独立验证。
- 有可能污染共享测试数据或外部系统。

提问时请包含：

- 观察到的行为。
- 期望行为。
- 证据。
- 影响。
- 它只阻塞当前用例，还是会影响整个套件。
- 推荐决策。

如果用户授权你在本次运行里自动修复未来出现的阻塞 BUG，就按这个偏好执行，并记录每次修复。

非阻塞 BUG 可以一边记录一边继续测试，除非用户要求“任何 BUG 都停”。

不要默默绕过阻塞 BUG。
