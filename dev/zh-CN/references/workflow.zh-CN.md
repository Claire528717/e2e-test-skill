# PRD E2E 测试代理工作流

当任务同时涉及用例设计和执行，或者用户希望更详细的 E2E 规划时，使用这份参考。

## 文档职责边界

保持职责分离：

- `SKILL.md`：只放强制运行规则和参考文件路由。
- `references/workflow.md`：放流程细节、依赖发现、评审门禁、执行就绪检查和结果处理。
- `references/coverage.md`：放 PRD 覆盖规则、状态定义和需求映射质量标准。
- `references/templates.md`：放输出结构和面向用户的审核包格式。
- `agents/reviewer.yaml`：放二轮复核角色、输入边界、检查清单和输出契约。
- `scripts/validate_plan.py`：放请求用户评审或执行前必须通过的确定性结构校验。

不要在这些文件之间重复长解释。能用模板字段、reviewer checklist 或 validator 校验约束的规则，优先放在对应约束点，而不是继续堆进 `SKILL.md`。
## 最佳适用场景

在前端/后端开发和联调完成之后使用。

适用于验收测试、发布回归、集成验证，以及创建、编辑、审批、订阅、上传、下载、授权、发布、取消发布、跨系统同步等流程。

不要把它当成单元测试、开发者自测、早期未完成联调排障、纯 API 自动化（没有 UI 路径）、性能测试、渗透测试或线上事故修复的替代品。

## 依赖发现顺序

在询问用户依赖细节之前，按这个顺序搜索：

1. 用户提供的 PRD、UI 资产和明确的测试目标。
2. 项目文档：README、部署文档、测试文档、API 文档。
3. 前端结构：路由、页面、菜单、按钮、表单、抽屉、弹窗、状态文本。
4. 配置文件：`.env`、`.env.*`、配置目录、代理、本地启动脚本。
5. API 和数据层：API client、服务封装、OpenAPI/Swagger、mock、fixture。
6. 现有测试：Playwright、Cypress、Jest、测试数据构建器、测试账号文档。
7. 数据库线索：ORM 配置、migration、schema、SQL、表名。
8. 外部系统线索：SDK、集成文档、环境变量、请求封装、日志。
9. 只向用户确认仍然缺失且必需的项。

在设计用例或执行之前，先汇总已发现的依赖和仍然缺失的依赖。

在询问用户之前，先判断每个缺失项能否通过以下方式合理处理：

- 阅读项目文档或源码。
- 使用本地启动脚本、fixture、seed 或 mock。
- 创建隔离的 mock 用户、角色、租户、权限、应用、文件或产品数据。
- 生成不包含敏感信息的测试文件。
- 将受影响用例降级为仅规划或标记为阻塞。

如果缺失依赖无法诚实解决，要直接说明。不要通过绕过 UI、编造凭证、写数据库模拟 UI 行为、伪造验证结果，或声称未验证结果已经通过来继续。

## 用户决策节点

按顺序使用这些节点：

1. **用例评审节点**：让用户评审测试用例质量、PRD 覆盖完整性、高风险标记、假设和统一数据需求。这个节点用于决定是否执行 E2E，不是最终上线验收，也不能证明执行环境已经就绪。
2. **阻塞 BUG 节点**：执行过程中，如果阻塞 BUG 会阻止当前用例、影响大量后续覆盖、阻止必要独立验证，或可能污染共享数据，就询问用户是否先修复再继续。
3. **结果与修复节点**：执行结束后，总结证据、BUG、清理状态、残余风险和延期项，让用户决定是否修复全部 BUG、延期部分 BUG、重跑用例，并查看修复结果。

在用例评审节点前，先生成 PM 审核包。审核包是用户主要阅读面，详细用例只是支撑材料。审核包必须包含决策摘要、自动质量门禁、二轮复核、覆盖摘要、阻塞项和当前需要用户拍板的事项。

## 执行就绪检查

用户明确要求执行 E2E 后，先刷新计划，再开始运行用例。确认当前环境能否：

- 打开目标 UI 或本地应用。
- 使用需要的账号登录，或者安全地创建这些账号。
- 完成所需的 UI 操作。
- 读取独立验证来源。
- 如果运行需要，创建、修改或清理测试数据。
- 避免对生产环境写入，除非用户明确批准。

如果有任何一项缺失，就更新测试计划中的依赖、数据和验证章节，然后返回依赖请求，不要暗示已经可以执行。

执行时要刷新每一条独立验证路径，因为用户此时可能已经提供了设计阶段没有的真实 URL、数据库、API、日志、外部系统访问或清理权限。能拿到具体信息时，要把“查询数据库/API”这类泛化表述替换为具体来源、查询 key、字段和断言。

当计划很大，或者属于 PRD 支撑的完整验收计划时，主 agent 必须通过可用的 multi-agent 工具先拉起一个独立的 reviewer 子 agent，再把计划交给用户。只给 reviewer 决策摘要、覆盖审计、mock/数据计划、阻塞项和 3-5 个最高风险用例，然后用 reviewer 的结果修订计划。对外只返回是否可执行、主要问题和用户现在必须拍板的事项。如果当前 AI 工具无法拉起或访问 reviewer 子 agent，就把二轮复核标记为阻塞或需要人工复核，说明原因，不能暗示已经完成独立复核。

主 agent 必须先规整 reviewer 输出再交给用户：必须修改项先修复，阻塞项移入决策摘要，非阻塞建议摘要化；对用户可见的 reviewer 结果只保留结论、必须修改项、用户决策项、理由和主 agent 处理结果。

本地测试环境通常不会自带完整的用户、角色、租户或权限。对于本地执行，如果项目支持，应优先创建或使用隔离的 mock 测试主体。如果项目无法安全创建或 mock 这些主体，就把受影响的执行标记为阻塞，并明确记录缺什么。

## 覆盖设计

按风险优先排序：

- **P0**：收入相关、关键流程、权限边界、持久化、外部集成、事实源一致性、或会阻塞发布的风险。
- **P1**：重要边界情况、替代状态流转、常见失败路径、或角色差异。
- **P2**：广泛回归或低风险覆盖；只有在用户明确要求时，或者套件有足够容量时才包含。

覆盖审计要求：

- 每个 PRD 支撑的完整验收计划都要包含 PRD 覆盖审计或需求覆盖矩阵。
- 如果 PRD 有章节号或稳定 checklist 标题，要保留这些定位信息。
- 每一行只能标记为已覆盖、部分覆盖、阻塞、超出 E2E/专项或超出发布范围中的一种。
- 高风险项要放在单独的风险列或高风险清单中；不要把风险或阻塞说明混进覆盖状态。
- 不要用一个相近 happy path 代替驳回、禁用态、角色差异、跨租户可见性、审计日志、防重复、重试、超时或失败状态。
- 对每个部分覆盖的高风险或 P0/P1 行，要补用例、把缺失分支补进已有用例，或在评审前记录具体缺失依赖。
- SLA、负载、可用性、安全等非 UI 要求要放进专项行，不要假装一次 UI 点击能验证。

每个用例都要包含：

- ID、优先级、标题、目的。
- 参与者/账号和所需权限。
- 前置条件和生成或需要提供的数据。
- 带有精确可见标签的 UI 操作路径。
- 后端/API/外部系统验证路径。
- 预期结果。
- 需要采集的证据。
- 清理方式。
- 阻塞判断规则。

## 测试数据规则

为每个套件使用一个 run ID，例如 `E2E-{feature}-{yyyyMMddHHmm}-{short_random}`。

数据分类如下：

- **测试主体**：账号、角色、租户、应用、归属、权限范围。
- **产品数据**：在被测产品内创建的记录。
- **外部数据**：在集成系统里创建的对象。
- **文件**：上传文件、OpenAPI 规范、资源包、图片或生成文档。

规则：

- 生成的数据都要带上 run ID 或 `E2E-` 标记作为前缀。
- 名称要唯一且可追踪。
- 对缺失的非敏感输入，只生成确定性的 mock 数据。
- 不要编造凭证、生产端点或审批身份。
- 当用户没有提供账号、租户、角色、权限、应用或文件时，根据 PRD 推导一份 mock 测试主体矩阵用于规划。
- 在环境能够创建或提供这些主体之前，把推导出来的主体当作规划需求，而不是现成可用的对象。
- 不要在真实执行阶段使用推导出来的主体，除非环境支持创建或 mock 这些主体。
- 记录 mock 范围：哪些用户、角色、租户、权限、应用、产品记录、文件和外部对象是 mock 的；为什么需要；如何创建；哪些用例会使用；如何清理或保留。
- 当用户要求新增、删除或修改用例、范围、角色、文件或验证方式时，必须在同一次修改中同步更新统一 mock/数据计划、测试主体矩阵、产品数据、文件、外部系统数据和清理规则。不要只改用例正文。
- 本地测试时，默认用户、角色、租户、权限和应用归属都应走 mock 路径，除非已经提供了真实测试主体。
- 保留失败或阻塞用例的数据，直到用户决定是否检查或清理。
- 对通过的用例数据，在安全前提下清理；如果不能清理，要写明原因。
- 不要使用真实的敏感个人、财务、凭证或生产客户数据作为 mock 数据。

## 评审门禁

在执行前，创建或更新一份专用测试文件，并包含：

- PM 审核包，包括决策摘要、自动质量门禁、二轮复核、阻塞项和用户决策。
- 范围和假设。
- 依赖发现结果。
- Mock / 数据计划。
- Mock 范围和假设，包括从 PRD 推导出来的内容。
- 账号、角色、租户、应用和权限的测试主体矩阵。
- 完整的可执行测试用例。
- 用户评审状态和需要修改的内容。

让用户评审这份文件的用例质量和完整性。如果用户修改了用例、数据、角色、验证方式或范围，就同步更新同一份文件中的统一数据计划；当这些修改会实质影响执行时，再次请求批准。文件很大时，先给决策摘要和主要问题，不要让用户为了做决定先通读全部用例。

执行结束后，把结果追加到同一份文件中：

- 整体结果摘要。
- 每个用例的状态：首次通过、修复后通过、未修复失败、阻塞、跳过。
- 证据引用。
- 发现的 BUG 和修复状态。
- 剩余需要用户决策的事项。
- 清理状态。

只有在用户要求，或者文件会大到不便阅读时，才使用单独报告。

## 面向用户的表达

对非技术用户使用直白表达：

- 用“我可以先设计用例，但现在还不能执行，因为……”来说明问题，不要只说技术黑话。
- 分开说明“我找到了什么”“我假设或 mock 了什么”“还缺什么”“现在还能测试什么”“需要你决定什么”。
- 不要用“环境问题”“依赖问题”这类模糊说法掩盖执行被阻塞。
- 使用 mock 数据时，要说明这是测试专用数据，并列出涉及的用户、角色、租户、权限、文件和记录。

## 术语

- **PRD**：产品需求文档，也是业务规则和验收标准的主要来源。
- **可视化 UI 来源**：HTML、原型、截图、运行中的产品，或用于识别真实操作路径的前端代码。
- **UI 操作路径**：通过页面、按钮、表单、抽屉、弹窗、上传和确认等可见界面完成的用户动作。
- **独立验证**：数据库、API、日志、队列、文件或外部系统中的证据，不能只依赖 UI 的口头结果。
- **阻塞 BUG**：会阻止当前用例或后续一大批用例继续执行，或者让后续证据不再可靠的缺陷。
- **非阻塞 BUG**：应该记录，但不会妨碍继续执行有价值的其他用例。
- **mock 数据**：不包含敏感真实信息的生成测试数据。
- **首次通过**：用例第一次执行就通过。
- **修复后通过**：用例第一次失败，随后修复了缺陷或环境问题，再跑通过。
- **未修复失败**：用例仍然失败，而且没有修复。
- **延期 BUG**：用户决定在本次运行中不修复这个 BUG。
- **事实源**：某类数据的权威系统，例如 API 资产的外部 API 中台。
