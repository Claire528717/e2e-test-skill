# PRD 覆盖规则

当设计或评审 PRD 支撑的 E2E 验收计划时使用本参考，尤其适用于用户询问测试套件是否覆盖 PRD、是否可作为交付测试标准的场景。

## 覆盖清单

在写最终用例前，优先从以下 PRD 来源抽取需求清单：

1. 明确 must-have / V1 范围清单。
2. 验收标准和发布阻断检查项。
3. 角色和权限矩阵。
4. 业务规则和可见性规则。
5. 状态机和不同状态下允许的操作。
6. 对象字段和数据一致性要求。
7. 异常、失败、空状态、禁用态、超时和重试状态。
8. 集成契约和外部系统副作用。
9. 版本、审计、清理和历史保留规则。
10. SLA、性能、可用性、并发等非功能要求。
11. 范围外或延期项，让评审人看到这些项是有意排除。
12. 未决产品问题和阻塞决策。

如果 PRD 有章节号，要保留章节号或稳定标题。如果 PRD 使用 checklist，除非多个项无法拆开验证，否则每个 checklist 项单独成为一行覆盖项。

## 覆盖状态

每个覆盖项只能选择一种覆盖状态。不要在一个状态单元格里混合多个状态；需要细节时使用单独的风险和决策字段。

- `已覆盖`：至少一个可执行用例通过可见 UI 路径和独立验证覆盖该项。
- `部分覆盖`：用例触及该项，但缺少角色、状态、失败分支、数据一致性检查或外部验证路径。
- `阻塞`：由于缺少产品决策、账号、权限、环境、事实源系统或测试数据路径，无法诚实规划或执行。
- `超出 E2E / 专项`：需要性能、压测、安全、可访问性、灾备或人工内容审核等其他测试类型。
- `超出发布范围`：PRD 明确排除或延期。

不要因为一个大流程从旁边经过，就把需求标为已覆盖。例如，审批 happy path 不覆盖驳回、重提、跨租户隔离、审计日志或禁用操作，除非这些分支被明确测试。

## 风险标记

高风险项必须和覆盖状态分开标记。

- 风险级别说明业务影响和执行优先级，例如 高 / 中 / 低 或 P0 / P1 / P2。
- 覆盖状态说明该项是否已经规划并可验证：已覆盖、部分覆盖、阻塞、超出 E2E / 专项或超出发布范围。
- 覆盖行不要使用 `已覆盖/阻塞待确认`、`已覆盖/部分覆盖` 这类混合状态。请选择一个覆盖状态，把风险、阻塞或决策写到 `风险` 或 `缺口 / 决策` 列。
- 高风险但部分覆盖或阻塞的项，要让用户不逐条阅读用例也能看见。

## 矩阵要求

完整交付验收计划必须包含以下任一章节：

- `## PRD Coverage Audit`
- `## PRD 全量覆盖审计`
- `## Requirement Coverage Matrix`
- `## 需求覆盖矩阵`

矩阵至少包含：

| PRD 项 | 验收点 | 风险 | 覆盖状态 | 用例 ID | 缺口 / 决策 |
|---|---|---|---|---|---|

对于高风险 PRD 计划，可以增加单独的高风险清单：

| PRD 项 | 风险 | 覆盖状态 | 用例 ID | 缺口 / 决策 |
|---|---|---|---|---|

## 用例扩展规则

对每个 `部分覆盖` 的高风险项，评审前执行以下动作之一。低风险部分覆盖项也必须明确缺失分支和所需决策：

- 如果共用同一套准备和验证，把缺失分支补进已有用例。
- 如果缺失分支有独立参与者、权限边界、状态流转、失败模式、事实源检查或清理规则，就新增用例。
- 如果无法执行，标记为阻塞或专项，并写清具体原因和用户决策。

除非 PRD 明确不要求，否则以下高风险分支应单独覆盖：

- 创建、编辑、删除/下架、发布、取消发布、审批、驳回、重提、转移、订阅、更新、下载、分享、撤销。
- 普通用户、领域/产业管理员、平台管理员、受限/跨租户用户、系统/集成主体之间的角色差异。
- 状态相关禁用操作，尤其是审核中、已驳回、已下架、已过期、未授权、已订阅和无数据状态。
- 数据主体差异，例如个人主体订阅和应用主体订阅。
- 外部副作用，例如 API Hub 同步、网关凭证颁发、资源包下发、审计事件、通知、队列任务或文件生成。
- PRD 要求明确提示的失败分支。

## 专项和阻塞项

不要把非 E2E 要求硬塞进弱 UI 用例。必要时在覆盖审计中标为专项或阻塞。

常见专项项：

- 页面响应 SLA、上传耗时、扫描耗时、凭证发放耗时、月度可用性、并发。
- 渗透测试和漏洞扫描。
- 可访问性审计。
- 数据迁移或灾备。

常见阻塞项：

- 安全扫描来源或规则未确认。
- 无法创建真实测试账号和角色。
- 缺少 API、数据库、日志或事实源访问。
- 缺少外部集成沙箱。
- 清理权限未定义。

要直白说明影响。例如：`安全扫描来源未确认，因此扫描不通过、重试、超时提示只能设计为用例，不能执行为上线证据。`

## 评审门禁

请求用户评审完整验收计划前，确认：

- 每个明确 PRD must-have 和验收项都出现在覆盖审计中。
- 每个已覆盖行都指向计划中真实存在的用例 ID。
- 没有覆盖行使用 `已覆盖/部分覆盖` 这类混合状态；风险和阻塞说明放在独立字段。
- 高风险部分覆盖或阻塞项在覆盖审计或高风险清单中可见。
- 每个 P0/P1 业务规则、权限边界、状态流转和集成副作用已覆盖或明确阻塞。
- 套件标题和范围诚实说明它是完整发布计划、聚焦切片、仅原型计划还是正式环境计划。
- 覆盖审计中不再残留已经新增用例后的 `建议用例` 表述。

如果这些检查未通过，先更新计划，再请求评审。
